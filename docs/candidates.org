#+title: linux-rg patch and policy candidates (rgx1gen11)

* Overview

This file documents carried modules, pending patch candidates, and userspace
policy options for the rgx1gen11 kernel profile. Literature grades refer to
=docs/literature-review.org=.

Carry rules from =docs/rgx1gen11-kernel.org= still apply:

- Public, reviewable source.
- Reproducible application from the PKGBUILD.
- Clean compile against Arch =7.0.12.arch1=.
- Behavior matches this ThinkPad profile.
- Disable or modularize when practical.

* Carried: ADIOS I/O scheduler

** Status

| Field | Value |
| Decision | carried (module, non-default) |
| Evidence grade | D (no peer-reviewed evaluation) |
| Patch | =0008-adios-cachy.patch= (extracted from CachyOS 6.17 omnibus) |
| Config | =CONFIG_MQ_IOSCHED_ADIOS=m=, =# CONFIG_MQ_IOSCHED_DEFAULT_ADIOS is not set= |

** What it is

ADIOS (Adaptive Deadline I/O Scheduler) is a multi-queue block scheduler for
=blk-mq= devices. It combines deadline scheduling with adaptive latency
control: it learns completion latency by operation type and size, predicts
completion time for new requests, and adjusts deadlines and batching within a
configurable global latency window.

Primary reference: https://github.com/firelzrd/adios

Version carried in the 7.0 extraction: 3.1.7 (CachyOS omnibus). Upstream ADIOS
3.2.0 exists for newer kernels via firelzrd stable patches; revisit on kernel
bumps.

** Why it is on this profile

The laptop uses a KIOXIA XG8 NVMe device under mixed interactive and build
load: browser caches, source trees, package builds, VMs, and sync tools share
one storage device.

BFQ remains the default because it has decades of published scheduler research
and is a conservative laptop default. ADIOS is carried as a module so block
scheduler choice can be tested without another DKMS or out-of-tree drift path.

** Literature context

- BFQ: Valente and Avanzini (ACM OSR 2012); evolution paper (Unimore).
- HOTSTORAGE 2023 (Whitaker et al.): on ultra-low-latency NVMe, mq schedulers
  often do not improve performance or energy efficiency and can increase
  latency. See =docs/literature-review.org=.

ADIOS has no peer-reviewed evaluation. Carrying it is an engineering option for
local measurement, not a literature-mandated addition.

** Relationship to other block choices

| Component | Role |
| BFQ (default) | cgroup-aware fairness; literature-backed for mixed workloads |
| Block contention patch | Reduces merge and dispatch lock contention in BFQ and mq-deadline |
| ADIOS (module) | Optional adaptive-deadline scheduler for A/B testing |

Do not assume ADIOS plus BFQ contention patch plus BFQ default is optimal.
HOTSTORAGE 2023 motivates comparing against =none= on this NVMe.

** How to test on the laptop

Switch scheduler on the NVMe queue (replace =nvme0n1= if needed):

#+begin_src sh
# show current scheduler
cat /sys/block/nvme0n1/queue/scheduler

# try ADIOS (module must be loaded)
sudo modprobe adios
echo adios | sudo tee /sys/block/nvme0n1/queue/scheduler

# revert to BFQ
echo bfq | sudo tee /sys/block/nvme0n1/queue/scheduler

# literature-motivated baseline from HOTSTORAGE 2023
echo none | sudo tee /sys/block/nvme0n1/queue/scheduler
#+end_src

Suggested measurement set:

1. Desktop feel: Sway, browser, editor, background =makepkg= or kernel build.
2. Storage latency: =fio= random read/write mixed workload on =/home=.
3. Energy: =powertop= or RAPL-backed logging during idle and mixed load.

Record: scheduler name, latency percentiles, interactive stutter subjectively,
and whether writeback flusher behavior changes under memory pressure.

** Sysfs tunables

Under =/sys/block/<dev>/queue/iosched/= when ADIOS is active:

- =adios_version= (read-only)
- =global_latency_window= (nanoseconds; lower tends to reduce latency)
- =bq_refill_below_ratio= (percentage of global window)
- =batch_actual_max= (read-only per operation type)

Start with defaults; tune only after baseline captures.

** Carry tradeoffs

- Adds =block/adios.c= (~2000 lines) and Kconfig integration.
- Conflicts possible with future Arch block-layer changes on rebases.
- Making ADIOS the default (=CONFIG_MQ_IOSCHED_DEFAULT_ADIOS=y=) is not
  literature-supported; keep BFQ default until local NVMe study completes.

** Decision gate for promoting to default

Promote ADIOS to default only if, on this machine:

- Interactive latency is consistently better than BFQ and =none= across repeated
  trials, and
- No regression in package build throughput or suspend/resume stability, and
- Energy is not worse than =none= at idle (HOTSTORAGE 2023 criterion).

* Candidate: re-swappiness v1.2

** Status

| Field | Value |
| Decision | candidate (not carried) |
| Evidence grade | D (patch); A (problem framing via ATC 2020 and SWAM) |
| Source | sirlucjan =7.0/re-swappiness-patches= |
| Upstream project | https://github.com/firelzrd/re-swappiness |

** What it is

re-swappiness replaces global swappiness semantics with MGLRU-aware reclaim
behavior. It adjusts how aggressively anonymous and file-backed pages are
scanned under pressure, using per-type generation tracking in the multi-gen LRU.

Touched files (typical 7.0 patch):

- =include/linux/mm_inline.h=
- =include/linux/mmzone.h=
- =mm/vmscan.c=
- =mm/workingset.c=

** Why it might help rgx1gen11

The profile enables MGLRU, zswap, zram, KSM, PSI, and DAMON. Under browser plus
build plus container load, reclaim policy strongly affects responsiveness.

Literature motivation:

- Lebeck et al., USENIX ATC 2020: default mobile memory management
  (swappiness and OOM killing) is often misaligned with responsiveness goals.
- Lim et al., SWAM (arXiv:2306.08345): revisiting swap and OOM killer policy
  for application responsiveness.

re-swappiness is a downstream implementation in that lineage, not an ATC 2020
or SWAM paper result itself.

** Risks and interactions

- Overlaps =mm/vmscan.c= with MGLRU mainline code and with =mglru-patches-v2=
  (another candidate). Do not stack without diff review.
- May interact with zram writeback and zswap pool pressure; test with the same
  userspace memory policy intended for production.
- No Lenovo or ThinkPad-specific validation found.

** Evaluation plan before carry

1. Baseline: current linux-rg with production zram/zswap systemd units.
2. Measure under fixed workload: browser tabs, =make -j9= kernel build, optional
   VM or container.
3. Capture: =/proc/pressure/memory=, swap in/out rates, OOM or near-OOM events,
   time-to-complete build, desktop latency subjectively.
4. Compare against mainline swappiness and =vm.*= tunables only (no patch).

** Decision gate

Carry only if responsiveness improves measurably without increasing swap thrash
or build time regression on repeated runs.

3. Run several suspend cycles on Sway to confirm no regression.

** Decision gate

Carry if TEO shows equal or better idle power without s2idle regressions. Skip

* Candidate: sched/fair detach_tasks loop_max fix

** Status

| Field | Value |
| Decision | carried (=0011-detach-tasks-fix.patch=) |
| Evidence grade | B (upstream-reviewed fix) |
| Source | cachyos-fixes-v27 patch 09/46; sirlucjan =7.0/cachyos-fixes-patches-v27= |
| vissue | =linux-rg-78xs= (DONE) |

** What it is

Fixes a fair-scheduler load-balance bug: =env.loop_max= in =detach_tasks()= was
set before acquiring the busiest runqueue lock, so the loop limit could disagree
with the actual =cfs.h_nr_queued= count under concurrent modification.

Patch behavior:

- Hold =env.loop_max= assignment until after =rq_lock_irqsave(busiest)=.
- Use =busiest->cfs.h_nr_queued= instead of =busiest->nr_running=.

Reviewed-by lines in the patch include Vincent Guittot and Valentin Schneider.

** Why it might help rgx1gen11

The i7-1365U is a hybrid 12-thread CPU. Load balancing across P-cores and
E-cores is frequent under mixed desktop and build workloads. Redundant
=detach_tasks()= scans waste CPU and can add latency jitter.

** Interaction with BORE

BORE also modifies =kernel/sched/fair.c=. This fix targets =sched_balance_rq()=
/ =detach_tasks()=, a different region from BORE's burst scoring. Likely
compatible, but an Arch-line-offset adaptation patch may be required (same
pattern as =0002-bore-fair-arch-adapt.patch=).

** Evaluation plan

1. Apply patch to a test build with BORE enabled.
2. Under =make -j9= plus desktop load, sample scheduler stats if
   =CONFIG_SCHEDSTATS=y= (enabled in overlay).
3. Optional: SpecJBB or similar load-balance stress if available; original
   report cited redundant scans in long runs.

** Decision gate

Carry if the patch applies cleanly with BORE and does not break the fair.c
build. Low functional risk; main cost is rebase and adaptation maintenance.

* Candidate: Cachy memory protection ratios (anon/clean min)

** Status

| Field | Value |
| Decision | candidate (not carried) |
| Evidence grade | B (concept); D (downstream extract) |
| Source | CachyOS =6.17/0004-cachy.patch= (extract; do not carry full omnibus) |
| vissue | =linux-rg-7zwz= |

** What it is

Three sysctl knobs for reclaim protection on each NUMA node (this laptop is
effectively one node):

- =vm.anon_min_ratio= -- hard floor for anonymous pages (default 1%). Kconfig
  help explicitly mentions zram overfill thrash.
- =vm.clean_low_ratio= -- best-effort floor for clean file pages (default 15%).
- =vm.clean_min_ratio= -- hard floor for clean file pages (default 4%).

Extraction scope (ratio-only, no =CONFIG_CACHY=):

| File | Role |
| mm/Kconfig | =ANON_MIN_RATIO=, =CLEAN_LOW_RATIO=, =CLEAN_MIN_RATIO= defaults |
| include/linux/mm.h | =sysctl_*= externs |
| kernel/sysctl.c | proc handlers for the three ratios |
| mm/vmscan.c | skip reclaim when node ratios would be violated |
| Documentation/admin-guide/sysctl/vm.rst | knob documentation |

Not required for ratios: Arch =CONFIG_CACHY=, Cachy fair.c tweaks, page-writeback
Cachy defaults, or swap cluster =#ifdef CONFIG_CACHY= blocks. Those stay out of
the extract.

** Why it fits rgx1gen11

The overlay enables zswap, zram, MGLRU, and PSI. Under browser plus build load,
anonymous reclaim into zram can thrash before PSI-driven userspace reacts. Ratio
knobs give a kernel-side floor aligned with SWAM and USENIX ATC 2020 framing
(see =docs/literature-review.org=).

** Weld with mglru-patches-v2

Complementary layers:

- Ratios: policy gates ("do not reclaim below X% on this node").
- mglru-patches-v2: MGLRU scan mechanics (evictable size, writeback handling,
  =try_to_inc_min_seq= refactor, batch eviction).

Both touch =mm/vmscan.c= but at different functions. A welded series should
apply mglru-patches-v2 first, then ratio hunks, then resolve any offset
conflicts in =shrink_lruvec= / =evict_folios= paths. Expect one Arch adapt
patch, not a blind stack.

** Weld with re-swappiness

Partial overlap on =mm/vmscan.c= and conceptually related (both tune MGLRU
reclaim). re-swappiness also changes =lrugen.max_seq[type]= per anon/file in
=mmzone.h= and =mm_inline.h= -- a direction mainline MGLRU may already mirror
in 7.0.12. Prefer mglru-patches-v2 plus ratios over re-swappiness plus ratios
unless re-swappiness alone wins a benchmark.

** Decision gate

Carry extracted ratio patch only if:

- Applies without =CONFIG_CACHY= and compiles on Arch 7.0.12.
- With zram active, reduces swap thrash under fixed browser/build load.
- Does not increase OOM frequency versus baseline in a repeated pressure test.

Not a memory patch. Adds =CONFIG_SCHED_CACHE= and scheduler logic to track a

** Status

** What Liquorix actually is (7.0.12-lqx1)

Liquorix is **not** a PREEMPT_RT kernel. It ships:

| Knob | Liquorix | Arch 7.0.12 (linux-rg base) |
| CONFIG_PREEMPT_RT | **off** | off |
| CONFIG_PREEMPT | voluntary (=y=) | voluntary (=y=) |
| CONFIG_PREEMPT_DYNAMIC | **off** (locked full preempt) | **on** (boot-tunable) |
| CONFIG_HZ | 1000 | 1000 (already) |
| CONFIG_SCHED_HRTICK | y | y (already) |
| Fair scheduler | **PDS** (=CONFIG_SCHED_PDS=y=) replaces EEVDF | EEVDF + BORE |
| Patch bulk | single ~130k-line zen omnibus | n/a |

Default CPU scheduler is **PDS** (Priority Deadline Skiplist), not BORE, not MUQSS,
not mainline EEVDF. Enabling PDS requires the full zen patch and conflicts with BORE,
0010-0013 fair.c work, and sched_ext coexistence testing.

* PREEMPT_RT (realtime kernel) -- what is worth taking?

** Status

| Field | Value |
| Decision | not carried |
| Evidence grade | A for control loops; **D** for this laptop daily driver |
| Source | linux-rt / =CONFIG_PREEMPT_RT= tree |
| vissue | =linux-rg-prert= |

** What it is

PREEMPT_RT converts most kernel spinlocks to sleeping locks and reduces non-preemptible
sections so SCHED_FIFO/RR deadlines are met. It is a **different kernel flavour**, not
a small patch on Arch PREEMPT.

Linux 7.x continues RT merge toward mainline, but Arch 7.0.12 desktop config does
**not** enable =CONFIG_PREEMPT_RT=.

** Why not on ThinkPad X1 Gen 11 daily driver

| Concern | Detail |
| Boot / driver coverage | iwlwifi, i915, SOF, NVMe paths have historically been RT lagging or fragile |
| Conflicts linux-rg stack | sched_ext + ASA + BORE tuned for !RT; rt-i915 (Cachy) needs RT and does not apply cleanly |
| Power / thermals | RT increases timer pressure; ultrabook idle power suffers |
| Use case mismatch | No hard real-time workload on this box (audio prod at most -> PipeWire + SCHED_FIFO userspace) |
| Boot safety | Highest regression risk of any option considered |

** What to use instead for "realtime feel"

| Need | Approach |
| Audio dropouts | PipeWire realtime module + user =@audio= group; not kernel RT |
| Low UI latency | BORE, 0010-0013, scx_lavd via scx_loader |
| Network jitter | BBR + fq; Wi-Fi power_save tuning (=linux-rg-iwl=) |
| Actual RT tasks | Separate RT VM or dual-boot linux-rt; never first linux-rg boot |

** Decision gate

Revisit PREEMPT_RT only for a dedicated benchmark partition or audio-prod experiment
with rollback, never as the primary rEFInd entry.

* Con Kolivas CK / MUQSS / BFS -- why not port?

** Status

| Field | Value |
| Decision | not carried; largely superseded |
| Last upstream release | 5.12-ck1 / MUQSS (April 2021); Kolivas retired CK/MUQSS Aug 2021 |
| Source | http://ck.kolivas.org/patches/ |

** What CK actually was

The CK patchset was a **full alternate scheduler class** (MUQSS: per-CPU
skiplist, earliest virtual deadline), not a set of fair.c tweaks. It replaced
CFS/fair scheduling wholesale on desktop-focused kernels (Arch =linux-ck=, etc.).
Goals: deterministic desktop latency, simple rules, no endless sysctl tuning.

MUQSS design explicitly evolved toward mainline EEVDF (Kolivas cites EEVDF/Staircase
Deadline lineage in =sched-MuQSS.txt=). Mainline 7.0 **is** EEVDF-based now.

** What linux-rg already takes from the Kolivas lineage

| Kolivas idea | linux-rg equivalent |
| MUQSS/BFS desktop interactivity | BORE burst scoring (fair-class tweak, not replacement) |
| SD scheduler =update_load_set= scaling | BORE patch cites Kolivas SD scheduler in =fair.c= comment |
| Smaller slices / lower migration cost | Partially mirrored in Cachy =CONFIG_CACHY= fair hunks -- not carried (BORE overlap) |
| Separate scheduler for low latency | =CONFIG_SCHED_CLASS_EXT=y= + scx_lavd / scx_bpfland / ASA router |

BORE is from Masahito Suzuki (firelzrd), not Kolivas, but the BORE fair.c patch
literally references Con Kolivas's SD scheduler for sysctl granularity logic.

** Why wholesale MUQSS port is a bad fit for 7.0.12 + rgx1gen11

1. **No maintained tree.** Last CK/MUQSS is 5.12 (2021). Forward-porting to 7.0
   is a multi-thousand-line scheduler rewrite with no upstream reviewer.
2. **Architectural conflict.** MUQSS adds =SCHED_MUQSS= and bypasses EEVDF fair
   class. linux-rg commits to EEVDF + BORE + sched_ext + ASA. You cannot stack
   MUQSS on top; you must *replace* the fair stack and drop BORE.
3. **Hybrid CPUs.** i7-1365U P/E topology post-dates MUQSS retirement. Kolivas
   design predates Intel hybrid load-balance work; mainline + scx experiments are
   the current answer for P/E placement.
4. **Kolivas's own conclusion.** Aug 2021 blog: ending MUQSS/CK maintenance;
   mainline had absorbed key ideas; further desktop work belongs in different
   approaches (community moved to gaming patches, scx schedulers).
5. **Better modern equivalents.** For MUQSS-like "desktop feel without ripping
   out EEVDF": try =scx_bpfland= or =scx_lavd= via scx_loader (sched_ext), or
   benchmark BORE (already carried) before attempting a dead scheduler port.

** If you still want Kolivas-style behaviour

| Approach | Cost | Recommendation |
| Port 5.12 MUQSS to 7.0 | Very high; unmaintained | Do not |
| Replace BORE with MUQSS branch kernel | Loses BORE, sched_ext coexistence testing | Only as separate experimental branch |
| Extract non-scheduler CK hunks from 5.12 | High rebase; likely upstream already has them | Only if a specific hunk is identified |
| scx_bpfland / scx_lavd on hybrid CPU | Userspace; reversible | **Try first** |
| cachyos-fixes #06-#08 inline hot paths | Low; compatible with BORE | Carried as =0010= |

** Decision gate

Revisit MUQSS only if BORE + sched_ext experiments fail on repeated hybrid-CPU
desktop trials *and* someone commits to maintaining a 5.12->7.0 scheduler forward
port. Otherwise treat Kolivas work as historical design input already partially
present in BORE and mainline EEVDF.

* CachyOS kernel patchset (CK / build / rust) -- deep dive

** Clarification: "CK" in CachyOS context

CachyOS does not ship Con Kolivas's historical CK/MUQSS patchset on 7.0.
The omnibus =0004-cachy.patch= (6.17) and sirlucjan 7.0 splits use
=CONFIG_CACHY= as a compile-time gate for Cachy-specific fair-class and
misc tweaks, not MUQSS. BORE in linux-rg already carries burst scoring
from the same author lineage and overlaps the fair.c region.

| Layer | Source (7.0) | Primary files | rgx1gen11 verdict |
| CONFIG_CACHY fair tweaks | Cachy 6.17 omnibus | kernel/sched/fair.c, sched.h | not carried; conflicts with BORE |
| Compiler ISA (x86-64-v3, znver4) | cpu-cachyos-patches | arch/x86/Kconfig.cpu, Makefile | not carried; 1365U is Intel, not znver4 |
| -O3 kernel build | kbuild-cachyos-patches | Makefile, init/Kconfig | candidate; Arch uses -O2 |
| ThinLTO distributed backend | kbuild-dev-patches | Makefile, scripts/Makefile.thinlto, vmlinux_a | candidate for cosmolab dist builds only |
| Rust KBUILD_RUSTFLAGS sync | omnibus + cpu patch | Makefile, arch/x86/Makefile | mirrors CFLAGS (-Copt-level, -Ctarget-cpu) |

** CONFIG_CACHY fair-class section (omnibus excerpt)

When =CONFIG_CACHY=y=, Cachy lowers EEVDF base slice and migration cost:

- =sysctl_sched_base_slice=: 350000 ns (vs 700000 mainline in that extract).
- =sysctl_sched_migration_cost=: 300000 ns (vs 500000).
- =SCHED_NR_MIGRATE_BREAK=: 8 (same threshold as PREEMPT_RT path).

These are desktop-latency policy choices without published evaluation (grade D).
linux-rg keeps BORE instead; do not enable =CONFIG_CACHY= fair hunks on this
profile.

** Why Rust is in the kernel (not a perf play)

Kernel Rust is **not** pushed because Rust is faster than C. The goal is:

| Goal | Mechanism |
| Memory safety | Type system + borrow checker eliminate large classes of UAF, double-free, and data races in *new* code |
| Safer driver authoring | =rust/kernel/= abstractions wrap C APIs; leaf drivers use safe APIs where possible |
| Incremental migration | New subsystems in Rust (binder IPC, Nova/Tyr DRM, rnull block, ...) without rewriting all of C |
| Security ROI | Industry consensus (Android, Google, Microsoft research) that kernel CVE reduction justifies the investment |

References: kernel =Documentation/rust/general-information.rst=; Rust-for-Linux
project; Greg K-H on permanent Rust support (2025); USENIX ATC 2024 Android
binder Rust evaluation.

On Arch 7.0.12 with =CONFIG_RUST=y=, the **active** Rust path on a desktop
build is mainly =CONFIG_ANDROID_BINDER_IPC_RUST=y= (binder IPC). Iris Xe uses
=CONFIG_DRM_I915= (C), not =CONFIG_DRM_NOVA= (Rust). So enabling Rust does
**not** rewrite the graphics, Wi-Fi, or NVMe stack on this laptop today.

** Cachy =KBUILD_RUSTFLAGS= is a separate, smaller question**

CachyOS syncs =KBUILD_RUSTFLAGS= with =KBUILD_CFLAGS= (-Copt-level, -Ctarget-cpu).
That only affects the **small fraction** of the image compiled from =*.rs=
(binder, optional Nova/rnull modules). It is build tuning for Rust objects, not
the reason upstream adopted Rust.

1. =CC_OPTIMIZE_FOR_PERFORMANCE_O3=: sets =KBUILD_RUSTFLAGS += -Copt-level=3=
   alongside =-O3= for C -- requires =kbuild-cachyos-patches= Kconfig symbol.
2. CPU target flags: =KBUILD_RUSTFLAGS += -Ctarget-cpu=x86-64-v3= tracks
   =-march=*= (cpu-cachyos-patches).
3. ThinLTO dist mode (kbuild-dev-patches): distributed LTO backend split.

** Overlay hygiene warning**

=rgx1gen11.config= currently sets =CONFIG_ZEN_INTERACTIVE=y= and
=CONFIG_CC_OPTIMIZE_FOR_PERFORMANCE_O3=y=. Neither symbol exists in mainline
7.0.12 Kconfig without Cachy patches. =scripts/config= may append them to
=.config=, but they are **inert** at compile time. Remove =ZEN_INTERACTIVE=
(entirely bogus). For =O3=, either carry =0001-Cachy-Allow-O3.patch= or drop
the overlay line and stay on Arch =-O2=.

* Carried: ASA sched_ext router (arXiv:2511.11628 prototype)

** Status

| Field | Value |
| Decision | carried (prototype) |
| Evidence grade | D (research paper; heuristic kernel port) |
| Patch | =0009-asa-sched-router.patch= |
| Config | =CONFIG_SCHED_ASA=y= (depends on =SCHED_CLASS_EXT= + =PSI=) |
| Userspace | =asa-router= -> =/usr/bin/asa-router= |

** What it is

Mixture-of-Schedulers (MoS) proposes an Adaptive Scheduling Agent (ASA) that
routes workloads to expert sched_ext policies instead of designing one monolithic
fair-class scheduler. The paper (Wang et al., arXiv:2511.11628) uses offline ML
(XGBoost ensemble) plus a three-stage calibration pipeline; runtime action loads
expert BPF schedulers via sched_ext.

linux-rg implements a kernel prototype faithful to the architecture split:

| Module | Implementation |
| Perception | PSI averages, runqueue load, context-switch rate -> sysfs |
| Decision | Heuristic classifier + time-weighted probability voting |
| Action | sysfs =recommended_scheduler= + uevents; =asa-router= calls =scx_loader= |

Default expert mapping (paper portfolio subset):

| Workload class | Default scx expert |
| interactive | scx_lavd |
| cpu | scx_rusty |
| io | scx_bpfland |
| mixed | scx_p2dq |

** Sysfs interface

=/sys/kernel/sched_ext/asa/=:

- =perception= -- sampled metrics (read-only).
- =workload= -- voted class name.
- =recommended_scheduler= -- mapped expert for current class.
- =mapping= -- =class=scheduler= table (read/write).
- =enabled= -- start/stop sampling timer.
- =cooldown_ms=, =sample_ms=, =voting_window=, =decay_pct= -- tunables.

** How to test

#+begin_src sh
# after booting linux-rg with CONFIG_SCHED_ASA
cat /sys/kernel/sched_ext/asa/mapping
echo 1 | sudo tee /sys/kernel/sched_ext/asa/enabled
watch -n1 cat /sys/kernel/sched_ext/asa/recommended_scheduler

# userspace action (needs scx_loader + scx-scheds)
sudo asa-router --enable --dry-run   # observe recommendations
sudo asa-router --enable             # live switching
#+end_src

** Limitations

- No XGBoost model shipped; heuristic classifier only until offline training
  pipeline from the paper is reproduced.
- Scheduler switch still requires userspace BPF load (sched_ext design).
- Not a replacement for BORE; operates at sched_ext layer when enabled.
