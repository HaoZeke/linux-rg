#+title: linux-rg literature review (rgx1gen11)

* Scope

This document records research-grade sources used to evaluate kernel choices for
the ThinkPad X1 Carbon Gen 11 profile. It complements the patch rationale in
=docs/rgx1gen11-kernel.org= and candidate notes in =docs/candidates.org=.

Hardware context:

- Intel Core i7-1365U, Raptor Lake-P/U hybrid (P-cores and E-cores).
- KIOXIA XG8 NVMe (ultra-low-latency class storage).
- Intel Iris Xe integrated graphics (display version below Lunar Lake).
- Intel CNVi Wi-Fi, Thunderbolt 4 docking.

Sources are limited to peer-reviewed venues, IETF drafts, kernel.org and LWN
primary coverage, USENIX/LPC proceedings, and vendor technical documentation.
Downstream patch aggregators are not treated as evidence on their own.

Companion: =docs/candidates.org= ecosystem survey and live hardware probe tables.

* Why kernel research feels absent ("why is no one studying this?")

If you tune a desktop kernel from Liquorix threads, CachyOS patches, or Kolivas
mailing-list lore, the honest answer is: **a lot of kernel work is never written
up as citable science**. That is not the same as "no one studies kernels." The
field is split across venues, incentives, and publication cultures that rarely
reach patch READMEs or distro forums.

** Structural reasons academia under-publishes *your* kernel questions

| Factor | Effect |
| Measurement cost | A scheduler or IO scheduler claim needs fixed hardware, kernel ABI, config matrix, thermal state, and workload script. One laptop model is one data point. Reviewers want generalization; practitioners want one box that boots. |
| Incentive mismatch | OS faculty moved toward cloud isolation, eBPF security, ML systems, and hardware-software co-design. "Does BFQ beat none on a ThinkPad NVMe under Sway?" is too specific for SOSP, too applied for theory venues. |
| Mainline as publication | Many "results" land as lkml threads + merge commits. EEVDF shipped in Linux 6.6 with LWN coverage, not an ACM proceedings paper. The *artifact* is the patch, not a PDF. |
| Industry owns production tuning | Meta (PSI, OOMD), Google (MGLRU path, Android binder Rust), Intel (hybrid scheduling LPC), and cloud vendors publish blogs and LPC slides. Peer review is optional when billions of machines run your code. |
| Downstream patch desert | BORE, Liquorix zen cuts, Cachy CONFIG_CACHY hunks, and ADIOS ship as git series with Phoronix posts, not controlled studies. Users confuse *availability* with *evidence*. |
| Negative results vanish | "mq-deadline did nothing on ULL NVMe" (HOTSTORAGE 2023) is valuable. Few academics publish "we tried ten schedulers on a laptop and nothing helped." |
| Reproducibility debt | Kernel + firmware + ACPI + panel + NVMe APST + iwlwifi power_save = opaque state. Papers that omit this are not reproducible on a second machine. |

| Reddit / Discord patch stacks | Engineering folklore; may be correct, unverified | D |

** The citation gap linux-rg sits in

Mainline Linux 7.0 on a 2023 ultrabook is **post-CFS, post-MGLRU-merge, mid-sched_ext**,
running **hybrid Intel** silicon that post-dates most classic scheduler papers.
The published literature lags the patch queue by 2-5 years. Your distro choices
therefore sit in a **known evidence desert**:

- Carried fair-class tweaks (BORE, 0010-0013): engineering bets, grade D.
- Carried zen MM/cpufreq cuts (0014-0016): Liquorix folklore, partially upstreamed.
- Literature-backed choices: MGLRU, PSI, BBR family, BFQ *class*, HOTSTORAGE
  skepticism on ULL NVMe defaults, hybrid scheduling *problem statement*.

That is why this repo grades patches A-D and separates *problem framing* (often A)
from *patch implementation* (often D).

** What would count as real science for this project

Minimum bar for promoting any candidate from D to B on this machine:

1. Fixed hardware: 21HM002GUS, documented firmware and BIOS.
2. Fixed workload scripts per =docs/rgx1gen11-targets.org= profiles.
3. A/B: stock Arch kernel vs linux-rg; never compare against a moving upstream tip.
4. Metrics: PSI averages, scheduler latency (perf sched), idle power (RAPL or
   powertop), NVMe fio percentiles, s2idle success rate -- not a single FPS chart.
5. Pre-declared success thresholds (already in targets.org acceptance tables).
6. Negative results recorded in =docs/candidates.org= decision gates.

This is reproducible systems measurement, not a paper -- but it is **more honest**
than most patch-stack marketing.

Takeaway: **the kernel is studied heavily for security, cloud scale, and protocol
measurement**. It is **under-studied for "my laptop feels snappier with patch X"**
where X is a downstream fair.c diff. That is the WTF feeling, stated precisely.

** EEVDF (mainline fair class since 6.6)

EEVDF (Earliest Eligible Virtual Deadline First) replaced CFS latency-nice heuristics
with an explicit eligible-time / virtual-deadline model. Primary sources are
kernel documentation and LWN pre-merge series (Molnar, Gorton, 2023-2024), not a
standalone ACM paper. Wikipedia and kernel docs summarize the algorithm; treat
design rationale as grade B, formal verification as absent.

- LWN: Replacing CFS with EEVDF (series)
  https://lwn.net/Articles/925371/
- Kernel documentation:
  https://docs.kernel.org/scheduler/sched-eevdf.html

Research implication: downstream patches that fight EEVDF fair.c (PDS, MUQSS,
large ZEN_INTERACTIVE hunks) are **opposing a mainline design with public review
history** but without a single "EEVDF beats CFS on laptops" paper. linux-rg stays
on EEVDF + BORE tweaks, not replacement.

** sched_ext publication trail

sched_ext adds a BPF-loadable scheduler class with safe fallback to fair.c.
Maintainer presentations at Linux Plumbers Conference 2024 are the authoritative
design record. Academic preprints on mixture-of-schedulers (ASA, arXiv 2025) build
on sched_ext but are not independent verification of scx_lavd/bpfland on consumer
hybrid CPUs.

- LPC 2024: The extensible scheduler class
  https://lpc.events/event/18/contributions/1667/
- LWN coverage:
  https://lwn.net/Articles/991205/

Evidence grade: B for infrastructure; D for any specific BPF policy until bench.

** Server-centric scheduler criticism (context, not prescription)

Lozi et al., EuroSys 2016 ("The Linux scheduler"; community name "decade of wasted
cores") documented socket-level scheduling inefficiencies on **large servers**.
OokCite DOI: 10.1145/2901318.2901326. The work influenced awareness of load-balance
bugs; it is **not** a laptop study and predates Intel hybrid topology. Useful for
understanding why detach_tasks-class fixes matter; do not extrapolate core-waste
numbers to a 12-thread ultrabook.

- https://doi.org/10.1145/2901318.2901326
- Author PDF: https://people.ece.ubc.ca/sasha/papers/eurosys16-final29.pdf

** Mixture-of-Schedulers / ASA (carried as kernel prototype)

- Wang et al., arXiv:2511.11628
  https://arxiv.org/abs/2511.11628

Proposes Adaptive Scheduling Agent (ASA): offline ML workload recognition,
machine-specific scheduler mapping table, runtime switching among sched_ext
expert BPF policies (scx_lavd, scx_rusty, scx_bpfland, etc.). Built on
sched_ext (Vernet, LPC 2024 -- grade B infrastructure).

linux-rg carries =0009-asa-sched-router.patch= as a heuristic prototype
(grade D until benchmarked on 1365U): kernel perception/voting sysfs plus
=asa-router= userspace action. Full paper ML pipeline not shipped.

** Page cache writeback (vm.dirty_*)

- Kernel documentation: https://docs.kernel.org/admin-guide/sysctl/vm.html

No peer-reviewed laptop study found for optimal =vm.dirty_ratio= on NVMe+btrfs.
